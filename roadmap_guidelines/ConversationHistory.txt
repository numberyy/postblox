The core features you'd want to replicate locally are: programmatic inbox creation, two-way email conversations, webhooks/events for incoming mail, threading, and optionally semantic search over messages.
Here's how I'd architect this for your setup:
Mail Server — Stalwart or Maddy
Stalwart is the strongest choice here. It's a Rust-based all-in-one mail server (SMTP + IMAP + JMAP) with a built-in REST API for account management. You can programmatically create mailboxes, and it handles SPF/DKIM/DMARC natively. Maddy (Go) is simpler but less feature-rich. Either runs as a systemd service on Fedora, fits right alongside Pulse.
API Layer — Elixir/Phoenix
Since you're already deep in BEAM land with Pulse, a thin Phoenix API that wraps the mail server makes sense. This becomes your "AgentMail API" — endpoints for creating agent inboxes, sending mail, fetching threads, searching. GenServer processes can hold per-agent state, and the BEAM's concurrency model is perfect for handling many agent mailboxes concurrently.
Event Pipeline — Webhooks via Sieve + Milter or JMAP Push
Stalwart supports JMAP push notifications and Sieve filtering. When an email arrives for an agent, you pipe the event into your Elixir app (either via JMAP EventSource, a milter, or a Sieve pipe-to script). From there, your Phoenix app can fan out to whatever agent needs to handle it — including Pulse if you want notification triage to pick it up.
Semantic Search — Optional Embeddings Layer
Store email bodies + embeddings in SQLite/PostgreSQL with pgvector. Index on ingest. This gives you the "search and reason over emails" capability that AgentMail offers.

Performance targets and what they actually mean:
Let me break this down into what our stack gives us for free vs. what we need to work for.
For idle memory, a Rust + Axum binary with a Postgres connection pool (5 connections) and no active work will sit at roughly 3-8MB RSS. This is just what the Tokio runtime, the allocator metadata, and the connection pool buffers cost. For comparison, a Node.js Express app idles at 50-80MB (V8 heap), and a Python FastAPI app at 30-50MB. Stalwart will add another 20-40MB idle. Postgres with shared_buffers at the default 128MB will use about 150MB. So the full stack (PostBox + Stalwart + Postgres) idles at roughly 200MB total, with PostBox being the smallest piece. On a 512MB VPS this is tight but workable. On 1GB it's comfortable.
For API latency, Axum on Tokio can serve ~100K requests/second on a single core for simple JSON responses. Our endpoints mostly do: parse request → query Postgres → serialize response. With a local Postgres and connection pooling, the database roundtrip is 0.5-2ms. JSON serialization for a message list is microseconds. So the realistic p99 for a cached read endpoint is 2-5ms, and for a write endpoint (insert + index) is 5-15ms. This is already faster than what any user or agent will notice — the network latency to reach the server dominates.
For email parsing, mail-parser processes real-world emails at roughly 5,000-10,000 messages/second on a single core, because it's zero-copy (it references slices of the input buffer rather than allocating new strings). This means inbound email processing will never be the bottleneck — even if you receive 100 emails/second (which would be extraordinary), the parser uses less than 2% of one core.
For IMAP sync (linked inboxes), each connected account holds one persistent TCP connection in IMAP IDLE state. This costs roughly 10-20KB of memory per connection (the TLS session state + read buffer). 100 linked accounts = about 2MB. Tokio handles thousands of idle connections without breaking a sweat — this is literally what it's designed for. The CPU cost is near-zero when idle (just keepalive pings every 29 minutes per IMAP RFC).
For semantic search with pgvector, query latency depends on index type and dataset size. With IVFFlat on up to 1M vectors (1M emails), query latency is 5-20ms. With HNSW it's 1-5ms but uses more memory. For a self-hosted personal/team tool, you're unlikely to exceed 100K emails, so this will always be fast.
For webhook dispatch, sending an HTTP POST to an external URL is I/O-bound. With Tokio and reqwest, you can easily dispatch 1000 concurrent webhook deliveries. The practical limit is the recipient's server, not PostBox.

The performance targets that actually matter for user perception are: first API response under 100ms (including TLS handshake), inbox list under 50ms, send email under 200ms (because SMTP submission to Stalwart takes time), and docker compose up to healthy in under 30 seconds.
What our current plan indicates:
The architecture is inherently performant because we're not doing anything expensive. PostBox is essentially a thin API layer over Postgres and Stalwart. The hot path is: HTTP request → SQL query → JSON response. There's no ML inference, no heavy computation, no large in-memory caches. The only potentially expensive operation is embedding generation for semantic search, and that's async (offloaded to Ollama or an external API, not blocking the request path).
The main performance risk isn't speed — it's memory growth over time. Specifically: if we cache parsed emails in memory instead of re-parsing from the database, memory grows linearly with usage. The fix is simple: don't cache. Parse on demand. mail-parser is fast enough that re-parsing a 50KB email takes microseconds. Disk is cheap, RAM is not (especially on VPSes).
The other risk is connection exhaustion — if linked inbox sync holds one IMAP connection per account and we don't cap the pool, someone with 500 linked accounts could exhaust file descriptors. The fix: configurable max concurrent syncs (default 50), with a queue for the rest.
The actual discipline that prevents bloat:
Honestly, the performance stuff is the easy part. Rust is fast by default. The hard part is architectural discipline. Here's what I'd commit to:
No feature flags that are never turned off. If something is always on, it's not a feature flag, it's dead code with an if statement around it.
No "framework" abstractions until the third time you need them. Write the concrete implementation first. When you have three things that look similar, then extract the trait.

Keep the dependency tree auditable. Run cargo tree regularly. If you can't explain why a dependency is there, remove it.

Pattern 2: SlopGate pushes to Pulse (real-time events)
MCP isn't designed for this. MCP is request-response — the client asks, the server answers. There's no "server pushes an event to the client unprompted." So for real-time notifications ("urgent email just arrived"), MCP alone isn't enough.
The options for real-time push:
WebSocket. SlopGate already has a WebSocket endpoint (planned). Pulse subscribes and gets events instantly. This is the most natural fit — Pulse keeps a persistent connection open, SlopGate pushes message.requires_action events as they happen.
Webhook. SlopGate POSTs to a local HTTP endpoint that Pulse exposes. Works but requires Pulse to run an HTTP server.
Unix socket / named pipe. Both are Rust, both are on the same machine. A Unix domain socket is the lowest-latency, simplest option for local IPC. No HTTP overhead, no port allocation.
D-Bus. Since you're on Hyprland/Sway and Pulse is a systemd service, D-Bus is already available. SlopGate publishes signals on org.slopgate.Events, Pulse subscribes. This is how KDE Connect works, how desktop notifications work, and how most Linux desktop IPC happens.
What I'd actually recommend for Pulse integration:
Use both patterns. They're not mutually exclusive.

MCP for pulling data on demand — overlay bar status, briefing content, inbox counts. Pulse calls slopgate.briefing every N seconds and renders it.
WebSocket (or D-Bus) for push — "urgent email from a VIP contact just arrived" shows as an immediate desktop notification through Pulse, not as a polling-delayed status update.
SlopGate's MCP server is already in the spec. The WebSocket endpoint is already in the spec. Pulse just connects to both. No special integration code needed in SlopGate — Pulse is just another client.
The one thing to think about later is whether Pulse should also expose itself as an MCP server to SlopGate — so SlopGate can call Pulse tools like pulse.notify("urgent email from john") or pulse.clipboard.write(email_body).
