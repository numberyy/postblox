# R7+ — Backlog

## Status

R7+ is the post-R6 backlog. Items here are not scheduled and not sliced.
This document captures intent and constraints so each item can be sliced
and scoped when picked up. Nothing in this file is a commitment; the
anti-bloat rules in `CLAUDE.md` still apply when any of these are pulled
into an active phase.

## Items

### Bitwarden as a `SecretStore` backend

**Scope.** Implement the existing `SecretStore` trait against the
Bitwarden CLI/API as a third backend alongside the OS keyring default
and the AES-GCM file fallback. No new public surface area in the trait.

**Constraints.**
- Trait stays single (`SecretStore`). Per `CLAUDE.md` dependency tier 2
  ("wrap, replaceable"), the keyring crate already sits behind the trait;
  Bitwarden joins the same boundary.
- No global state. All handles owned by the daemon and passed explicitly
  per `CLAUDE.md` § "No Global State".
- Honor the existing AES-GCM file fallback; selection between backends is
  config-driven, not compile-time.
- No `lazy_static!` / `once_cell` globals; bounded channels only if any
  background task is required for unlock/refresh.

### LLM-gateway-driven search

**Scope.** Allow an LLM gateway to perform hybrid keyword + semantic
ranking server-side, reading from the same SQLite file that the daemon
owns. Either via direct read-only file access or via a new IPC tool
surface that reuses `search.query`.

**Constraints.**
- Must not break the existing FTS5 path used by the TUI and MCP bridge.
- Daemon stays the single writer. Any gateway access is read-only.
- `sqlite-vec` is already in the decision log (2026-05-06); the gateway
  ranker consumes the same index, it does not replace it.
- IPC bounds (4 MiB frame, 64 connections, 32 subs/conn) still apply if
  exposed via the existing socket.

## Open questions

- **Bitwarden auth lifecycle.** CLI `bw unlock` session token vs API key
  vs personal access token. Pick when picked up; affects whether we need
  a refresh task and how we surface re-auth in the TUI.
- **Gateway search transport.** Direct read-only file open from the
  gateway process, a new `search.*` IPC op, or a localhost HTTP endpoint.
  Pick when picked up; trade-offs are coupling, auth surface, and whether
  the gateway is colocated with the daemon.

## Anti-bloat reminders

- Do not start either item until a third concrete need exists. Two
  speculative use cases are not enough — `CLAUDE.md` "no abstractions
  before the third use" applies to backlog work too.
- No abstractions speculatively. The Bitwarden backend slots into the
  existing trait; no new layers above it.
- Every PR must be usable by itself — no trait-shaped scaffolding ahead
  of the concrete implementation.
- Delete what isn't tested. Backlog items land with tests or they don't
  land.
- Dependencies earn their place: each new crate needs a one-sentence
  justification and must be MIT/Apache-licensed.
